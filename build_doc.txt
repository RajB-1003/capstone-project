Architectural Blueprint for a Production-Grade Semantic FAQ Search System
System Architecture and Modular Decoupling
The deployment of a production-grade Semantic Student FAQ Search System necessitates a rigorous, decoupled architectural paradigm.1 Monolithic scripts, where user interface rendering, routing logic, and database operations are intertwined, inherently fail at scale, introduce severe technical debt, and prohibit concurrent team development.2 To ensure a scalable, future-ready Retrieval-Augmented Generation (RAG) framework, the project must be strictly partitioned into discrete, highly specialized modules.2
The architecture must be separated into the following clear production-level modules, establishing strict data contracts between them to ensure that a modification in one module does not initiate cascading failures across the system:
Data Ingestion: This module is responsible for parsing raw FAQ inputs, extracting critical metadata (e.g., category, intended audience, applicable semester), and structuring the text into optimal chunk sizes.1 Proper ingestion ensures that downstream retrieval algorithms operate on high-fidelity data, preventing the dilution of semantic meaning that occurs when documents are poorly segmented.
Embedding Pipeline: The embedding pipeline translates human-readable text into dense numerical vectors.1 It requires robust normalization techniques and consistent pooling strategies (typically average pooling) to ensure that the semantic representation is mathematically stable across varying query lengths.6 For local execution, this pipeline must rely on efficient models, such as sentence-transformers/all-MiniLM-L6-v2, to maintain a minimal memory footprint.
Vector Storage: The vector store acts as the persistence layer for the generated embeddings.1 For a constrained dataset of 20–50 FAQs, local implementations such as FAISS (Facebook AI Similarity Search) or ChromaDB are mandated.7 The vector storage module must support metadata filtering, allowing the system to execute pre-filtering prior to performing mathematical similarity calculations, thereby enhancing both accuracy and computational efficiency.
Retrieval Logic: This layer encompasses the algorithms utilized to fetch relevant documents from the underlying storage.1 It must support both sparse retrieval (e.g., keyword matching algorithms like BM25) and dense retrieval (e.g., semantic vector searches utilizing cosine similarity).5 The retrieval logic operates entirely agnostic of the user query's origin, functioning purely as a computational engine that accepts a query and returns ranked documents.
Agentic Router: Functioning as the central orchestration nervous system, the router dynamically analyzes incoming queries, detects underlying intent, and dictates the control flow of the application.1 It is the most complex decision-making layer, determining whether to invoke sparse retrieval, dense retrieval, or a mathematically fused hybrid approach.11
UI Integration: The user interface layer, built using Streamlit, must remain purely presentational.1 It interacts with the backend strictly through defined application programming interfaces or typed Python objects, rendering the final answer, the retrieved context, and the explainability metrics without ever executing routing or retrieval logic natively.12
Role Definition: The Agentic Router Module
Within a distributed development team, the member assigned to the Agentic Router assumes the critical role of system orchestrator.10 The router acts as the intelligent boundary between the unstructured user query and the rigid mathematical retrieval systems.1
The responsibilities inherent to this role extend far beyond writing conditional routing statements. The developer must construct an intelligent execution layer capable of parsing complex natural language, categorizing intent with sub-millisecond latency, and executing tools dynamically.11 The primary mandate is to improve retrieval accuracy by ensuring that highly specific queries (e.g., "What is the prerequisite for CS-202?") are not lost in the probabilistic noise of semantic search, while simultaneously ensuring that conceptually vague queries (e.g., "exam rules") are not met with zero results by a rigid keyword parser.1
To fulfill these requirements, the router module must maintain specific data contracts with both the Data/Database team and the Frontend UI team. From the Data team, the router expects exposed retrieval functions (search_keyword, search_semantic) that return standardized document structures. To the UI team, the router must guarantee a structured payload containing the routing rationale, the final retrieved documents, and normalized similarity scores necessary to render the "Why this result?" explainability panel.1 The router developer is thus responsible for state management, workflow orchestration, and the mathematical synthesis of search results.

Evaluating Router Decision Logic: Rule-Based vs. LLM-Based Routing
A critical design choice in building an Agentic Router is determining the mechanism for query classification. A prevalent, yet highly inefficient, approach is the naive reliance on a Large Language Model (LLM) to route every incoming query.17 The LLM is prompted to analyze the query and output a JSON object dictating the required tool.17
This LLM-based routing approach is fundamentally flawed for a system constrained by a strict sub-2-second local execution budget.1 Routing via an LLM introduces severe latency penalties (frequently ranging from 500 to 2000 milliseconds for a single inference pass), incurs high token generation costs, and introduces probabilistic non-determinism where the model may hallucinate incorrect tool names or fail to format the JSON output correctly.17 For a small, local FAQ system, invoking an LLM simply to decide between keyword and semantic search represents a massive over-allocation of computational resources.17
To achieve high performance, scalability, and strict latency compliance, the architecture must implement a multi-tiered, hierarchical routing logic that prioritizes fast, deterministic rules before defaulting to probabilistic methods.11
Tier 1: Deterministic Regex Scanning for Entity Detection
The primary objective of the router is to identify queries that require exact-match retrieval.1 Standard semantic embedding models struggle significantly with specific alphanumeric identifiers, such as course codes, building numbers, or fee receipt IDs, because these tokens lack semantic meaning and are frequently split into meaningless sub-word fragments by the tokenizer.22
Therefore, the first layer of the routing logic must be a deterministic Regular Expression (Regex) scanner.19 The system pre-compiles a dictionary of regex patterns representing known entity structures within the university context.19
When a query is received, it is processed through the Regex engine. If a pattern match is detected (e.g., the system detects the pattern [A-Z]{2,3}-\d{3} representing a course code), the router instantaneously classifies the query as an exact-match request and directs the execution flow directly to the keyword_search tool.1 This deterministic evaluation executes in approximately 1 to 5 milliseconds, entirely bypassing the latency of embedding generation or LLM inference.19
Tier 2: Vector-Based Semantic Intent Classification
If the query passes through the Regex scanner without triggering a match, the system must deduce that the query is conceptual or natural language-based (e.g., "What happens if I fail a class?").1 Rather than forwarding this query to a heavy LLM for intent classification, the router should employ a high-speed embedding-based intent classifier.20
In this mechanism, the system maintains a small, pre-embedded index of "intent exemplars"—a set of 10 to 20 highly representative queries mapped to specific search strategies.20 The incoming query is embedded using a lightweight model, and its cosine similarity is computed against the intent exemplars.20 If the query exhibits high similarity to an exemplar associated with conceptual policies, the router immediately directs the flow to the semantic_search tool.1 This vector-based routing approach guarantees low-latency intent resolution without the overhead of an LLM.24
Tier 3: Hybrid Search Fallback
There are edge cases where a query contains both a specific entity and a complex conceptual question (e.g., "Are there any scholarship prerequisites for enrolling in ENG-404 next semester?"). In these scenarios, the Regex scanner will detect the course code, but the semantic classifier will also detect high intent for policy retrieval.1
When the router detects conflicting signals, or when the confidence scores from the semantic intent classifier fall below a predetermined threshold (e.g., a cosine similarity score below 0.65), the system must default to the hybrid_search tool.1 Hybrid routing acts as the ultimate safety net, ensuring that neither the precise keywords nor the broader semantic context is ignored.1

Sparse Retrieval via BM25
For queries directed to the keyword search tool, standard token overlap metrics are insufficient. The architecture must utilize the BM25 (Best Matching 25) algorithm, an advanced iteration of the TF-IDF (Term Frequency-Inverse Document Frequency) framework.5 BM25 excels at exact keyword matching by heavily penalizing excessively long documents and adjusting term weights based on corpus frequency.9 This ensures that common words (e.g., "university", "student") do not skew the relevance scores, while highly specific identifiers (e.g., "CS-101") dominate the ranking.9
Hybrid Search Ranking Strategy: Reciprocal Rank Fusion
When the router dictates the use of the hybrid_search tool, the system faces a severe mathematical discrepancy.1 Sparse algorithms (like BM25) and dense algorithms (like Cosine Similarity) output scores on entirely incompatible scales.25 BM25 scores are unbounded, theoretically reaching to infinity, while Cosine Similarity scores are strictly bounded between -1.0 and 1.0. Directly aggregating or averaging these raw scores is mathematically invalid and will cause the BM25 results to completely overwrite the semantic results.25
To orchestrate a highly accurate hybrid search, the architecture must implement Reciprocal Rank Fusion (RRF).25 RRF elegantly bypasses the scale discrepancy problem by discarding the raw computational scores entirely, relying solely on the rank position of the documents within their respective lists.25

LangGraph Workflow Orchestration
To transition the routing logic from isolated functions into a production-grade execution layer, the system must utilize LangGraph to construct a deterministic state machine.3 Unlike traditional LangChain chains that execute sequentially, LangGraph allows for cyclical execution, complex branching, and strict state management, preventing the agent from descending into infinite hallucination loops.14
The implementation involves defining a typed state object that persists throughout the graph's execution. This state dictionary must include keys such as query (the raw input), detected_entities (extracted IDs from Regex), route_decision (the output of the intent classification), and retrieved_context (the final output of the search tools).2
The workflow is constructed using discrete nodes. The initial node acts as the Router, executing the Regex and semantic classification algorithms previously defined.33 The output of this node dictates a Conditional Edge. The conditional edge acts as a programmatic switchboard; based on the value populated in the route_decision state variable, the edge explicitly routes the execution to one of three downstream nodes: Node_Keyword, Node_Semantic, or Node_Hybrid.11
This graph-based architecture guarantees that the execution flow remains entirely within developer control.10 If an anomaly occurs, the system's state can be inspected at any discrete node, providing deep observability that is impossible to achieve with a black-box LLM router.2
Architecting the Explainability Layer: "Why this result?"
Providing a highly accurate retrieved document is insufficient for modern AI systems; user trust hinges on the system's ability to explain why a specific document was deemed relevant.1 The Agentic Router module must export structured rationale to power the "Why this result?" visualization panel in the Streamlit interface.1
The Explainability Layer requires two distinct modes of technical implementation: structural reasoning and token-level similarity visualization.
Structural Routing Rationale
The easiest path to explainability is surfacing the deterministic routing decisions made by the state machine.1 The state object must compile a natural language justification logging the exact path the query took.1
For instance, if the Regex scanner triggers, the router exports a payload stating: "Exact Keyword Search deployed due to the detection of a structural identifier (Course Code: CS-101)." If the semantic classifier triggers, the payload states: "Semantic Search deployed. Concept match established between query 'test policy' and document 'examination rules' with a confidence interval of 82%.".1 This level of transparency immediately demystifies the retrieval process for the end user.35
Token-Level Similarity Score Mapping
For semantic and hybrid searches, presenting a raw cosine similarity score (e.g., "Cosine Similarity: 0.814") is overly technical and provides poor user experience.1 The system must transform these opaque metrics into "glass box" visualizations by highlighting the exact words within the retrieved FAQ that contributed most heavily to the match.6
Implementing this requires deep integration with the embedding model's tokenizer.6 The technical process involves several rigorous steps:
Query Embedding: The user's query is embedded into a single, unified vector.6
Passage Tokenization: The retrieved FAQ document is tokenized. Instead of pooling the entire document into a single vector, the system extracts the high-dimensional hidden state vector for every individual token.6
Word-Level Aggregation: Because tokenizers frequently split single words into multiple sub-word fragments, the system must use the tokenizer's span maps (word_to_chars or word_ids) to mathematically average the token vectors back into cohesive word-level embeddings.6
Similarity Matrix: The unified query vector is multiplied against the matrix of word-level embeddings using a vectorized dot product operation.6 This generates a distinct similarity score bounded between 0.0 and 1.0 for every single word in the retrieved FAQ document.6
Payload Export: The router compiles these scores into a structured JSON payload, mapping exact character spans to their respective similarity scores.6
The UI team then consumes this payload within Streamlit, applying dynamic CSS styling to render words with high similarity scores in bold, dark text, while fading irrelevant connective words into a light gray.6 This visualization instantly proves to the user why a specific document was selected, even if the exact vocabulary differs from the original query.6
Caching Strategy and Strict Latency Optimization
The entire semantic search framework is operating under a severe constraint: it must run locally and return results in under 2 seconds.1 Local, CPU-bound execution environments are inherently hostile to deep learning operations.1 To prevent widespread latency breaches, the architecture must implement aggressive caching strategies and strict execution profiling.1
Embedding Pipeline Constraints
Generating embeddings is the single most computationally expensive operation in the system.1 It is an absolute architectural requirement that the embeddings for the 20–50 FAQs are generated exactly once during the initial data ingestion phase and persisted to the local disk via the vector store.1 Under no circumstances should document embeddings be calculated dynamically at runtime.
LRU and Semantic Query Caching
To further optimize performance, the system must cache incoming queries. The architecture should implement an In-Memory Least Recently Used (LRU) cache at the very top of the routing pipeline. If a query is an exact string match to a previous query (e.g., "Where is the library?"), the system immediately returns the cached payload, dropping latency to near-zero milliseconds.
For a more advanced optimization, the system can deploy Semantic Caching. When a new query is embedded, the system checks the vector store of previously asked queries. If the cosine similarity between the new query and a cached query exceeds an exceptionally high threshold (e.g., > 0.98), the system assumes identical intent and returns the cached result, entirely bypassing the underlying vector search and RRF mathematical operations.37
Latency Budget Allocation
To guarantee compliance with the 2-second constraint, the system must be profiled against a rigid latency budget.1 The execution path must adhere to the following metrics:
Regex Classification: Must execute in under 5 milliseconds via pre-compiled RE2 engines.19
Query Embedding: Must complete in 100–300 milliseconds. This requires the use of lightweight models (e.g., 384 dimensions) rather than massive 1536-dimension API models.37
Vector Search & BM25: For a small corpus of 50 documents, keeping the FAISS index and BM25 matrix entirely in RAM guarantees retrieval times under 10 milliseconds.8
Reciprocal Rank Fusion: Vectorized mathematical fusion utilizing NumPy must execute in under 5 milliseconds.30
By strictly profiling the execution layer and enforcing deterministic routing algorithms over massive LLM inference calls, the Agentic Router achieves the required stability, accuracy, and extreme low latency mandated by a production-grade specification.
